# 端到端发布工作流 {#release-workflow}

本页介绍了由版本管理器管理的协调版本中涉及的全序列活动。

## 1.为每个应用程序定义功能标志 {#define-flags}

每个产品团队都分配一个&#x200B;**产品版本所有者**，该所有者登录到“标记”控制台并为他们拥有的应用程序创建功能标记。 然后，产品团队在其代码中实施条件逻辑，将功能置于这些标记之后。

## 2.创建版本 {#create-release}

创建新版本需要支持请求 — 它并非完全自助。 联系标记支持以创建版本。 提供版本名称、所属团队、目标环境、目标、参与应用程序和预期持续时间。

确认发布后，发布管理器将打开控制台并完成发布配置。

## 3.为版本添加功能标记 {#add-flags}

对于添加到此版本的每个应用程序，该应用程序的产品版本所有者将登录并向版本中添加相关的功能标记。 这些标记表示将在版本中启用的功能。

## 4.配置受众 {#configure-audience}

发布管理器为该版本选择受众 — 例如，特定国家/地区的1%的用户、具有特定电子邮件域的所有用户或特定用户ID列表。 配置完毕后，版本管理器将保存并发布版本，以便让该版本对指定受众生效。

## 5.扩展和管理转出 {#expand}

在版本正式发布后，版本管理器可以调整受众规则以逐步扩展推出、监控问题并使用版本状态控件管理生命周期。 有关详细信息，请参阅[发行状态](release-states.md)。

## 要点 {#key-points}

* 所有参与的应用程序都将其代码独立部署到功能标记后的生产环境中 — 不需要协调的部署时间。
* 此版本是一个单点控制，可同时跨团队激活所有标记。
* 发行版的受众定位基于身份验证令牌属性（国家/地区、电子邮件、用户ID、百分比）。
